# Data files

Computable data files consumed by the skills under [`.claude/skills/`](../.claude/skills/) and by the scripts under [`scripts/`](../scripts/). Each file is intended to be:

- **Single source of truth** — referenced from skills/scripts rather than duplicated inline.
- **Shareable** — easy to cite, import into other curation projects, or load with standard tools (`pandas.read_csv`, `csv.DictReader`, etc.).
- **Validatable** — every ontology CURIE in any crosswalk here passes `runoak info` + `runoak ancestors -p i` against its slot's anchor class.

Two kinds of file live here: **hand-curated artifacts** (the crosswalks — deliberate curation, not pipeline output) and **generated-but-pinned inputs** (the habitat table and the vendored ontology — deterministic, regenerable, pinned to an upstream commit). The table below marks which is which.

## The two-tier crosswalk

`isolation_source` on the NCBI biosamples is a lossy MFD **Level-1** projection (e.g.
`"Soil from Natural Dunes"`), which is why a single-tier crosswalk leaves ~2,586 env-triad
slots as sentinels. The two-tier design resolves a biosample primary-first:

1. **Primary (L2)** — biosample `MFDID` → `mfd_habitat_per_sample.tsv` → habitat 5-tuple
   `(sampletype, areatype, hab1, hab2, hab3)` → `mfd_envo_crosswalk_l2.tsv`, matched at the
   deepest available habitat depth.
2. **Fallback (L1)** — biosample `isolation_source` → `mfd_envo_crosswalk_l1.tsv`, for the
   ~23% of biosamples whose `MFDID` is absent from the habitat table.

**Pipeline order:** `build_mfd_habitat_table.py` → `build_mfd_envo_crosswalk_l2.py` →
`apply_mfd_env_triad.py`. See [`scripts/README.md`](../scripts/README.md).

**L1 ⇄ L2 consistency:** the depth-1 (hab1-level) rows of L2 and the rows of L1 are the
same hab1-level crosswalk keyed differently; they must agree. When a curation establishes
a better hab1-level term, update both.

## Conventions

- **`ENVO:00000000` is the refuse sentinel** — any `*_curie` column with this value means "no ENVO term exists under the slot's anchor class for this row." The agent leaves the sentinel and proceeds to `nmdc-env-triad.md` §1b inference for that slot. Mirrors the deterministic pipeline's sentinel convention in `translate.py`. The corresponding `*_label` column is empty.
- **Anchor classes** — every committed CURIE descends (via `runoak ancestors -p i`) from its slot's anchor: `env_broad_scale` → `ENVO:00000428` (biome); `env_local_scale` → `ENVO:01000813` (astronomical body part); `env_medium` → `ENVO:00010483` (environmental material).
- **CURIE format** — `^ENVO:\d{8}$` for every ENVO term (regex-checkable).
- **L2 `curation_status`** — `carried_l1` (hab1-level triple inherited unchanged from L1), `refined` (a hab2/hab3 curation changed ≥1 slot), `new` (combo absent from L1 — should be 0 after curation).
- **Sort order** — `mfd_envo_crosswalk_l1.tsv` is sorted by `count_observed` descending; `mfd_envo_crosswalk_l2.tsv` is sorted by `(sample_type, area_type, hab1, hab2, hab3)` so habitat families are contiguous.
- **No tabs in field values.** TSVs are unforgiving — keep notes and labels free of tab characters.
- **ENVO release** — CURIEs were validated against ENVO via oaklib `sqlite:obo:envo` on 2026-05-19. Re-validate on ENVO releases (deprecations/label changes become bugs to fix here).

## Maintainer rules

1. **No CURIEs from memory.** Every ENVO CURIE must come from a `runoak search` lookup validated with `runoak info <CURIE>` (exists, label correct, not deprecated) and `runoak ancestors -p i <CURIE>` (the slot's anchor class appears). Mirrors `nmdc-curation-rules.md` Rule 6.
2. **L2 edits go to the generator.** Refinements to `mfd_envo_crosswalk_l2.tsv` go to the `CURATION` table in `scripts/build_mfd_envo_crosswalk_l2.py` (with runoak evidence in comments), then regenerate. L1 edits go to `mfd_envo_crosswalk_l1.tsv` directly. Keep L1 and L2 depth-1 consistent.
3. **Schema bumps require re-validation.** When ENVO publishes a new release, re-run the validators on every CURIE in both crosswalks — carried-forward terms included.
4. **Generated files are regenerable.** `mfd_habitat_per_sample.tsv` and `mfd_habitat_ontology.xlsx` are pinned to a cmc-aau/mfd_metadata commit; rebuild via `build_mfd_habitat_table.py` rather than editing by hand.

## Loading examples

```python
# Python — the L2 primary crosswalk
import csv, pathlib
with pathlib.Path("data/mfd_envo_crosswalk_l2.tsv").open() as f:
    l2 = list(csv.DictReader(f, delimiter="\t"))
# l2[0] is a dict keyed by column name
```

```bash
# shell — L1 rows for one sample type
awk -F'\t' 'NR==1 || $1=="Soil"' data/mfd_envo_crosswalk_l1.tsv
```

```python
# pandas
import pandas as pd
l2 = pd.read_csv("data/mfd_envo_crosswalk_l2.tsv", sep="\t")
```
